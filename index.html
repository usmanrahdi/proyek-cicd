🤔 CI/CD itu apa sih?

Bayangkan kamu jualan online. Setiap kamu update foto produk, kamu harus upload manual ke 5 toko berbeda. Melelahkan kan?

CI/CD itu seperti punya asisten yang otomatis menyebarkan updatemu ke semua toko setiap kali kamu simpan perubahan — plus ngecek dulu apakah fotonya tidak rusak sebelum dikirim.

💡 CI (Continuous Integration) = pengecekan otomatis setiap kamu simpan kode baru.
💡 CD (Continuous Deployment) = pengiriman otomatis ke internet setelah lolos cek.

🛠 Tools yang kita pakai & alasannya

Git — alat pencatat setiap perubahan kode. Seperti "riwayat versi" di Google Docs, tapi untuk kode.

GitHub — tempat menyimpan kode di cloud (gratis). Semua developer profesional pakai ini.

GitHub Actions — fitur bawaan GitHub untuk otomasi. Kamu tulis instruksi, GitHub yang jalankan.

GitHub Pages — hosting gratis dari GitHub. Websitemu bisa online tanpa bayar server.

📅 Cara pakai modul ini:
Sesi 1–3 adalah fondasi, sebaiknya dikerjakan urut. Sesi 4–7 bisa lompat hari. Semua progress tersimpan otomatis di browser — centang task yang sudah selesai, dan itu tetap ada walau kamu tutup tab.

A · Install Git di Mac
1
Buka Terminal
Tekan Cmd + Space di keyboard → ketik Terminal → tekan Enter. Kamu akan melihat layar hitam/putih dengan teks. Jangan panik — ini normal!
🖥 Kamu akan lihat teks seperti: nama-mac ~ % atau nama-mac:~ user$
2
Cek apakah Git sudah ada
Ketik perintah berikut lalu tekan Enter:
% git --version
Kalau muncul sesuatu seperti git version 2.xx.x → Git sudah ada, skip ke langkah 4.
Kalau muncul pop-up "Install Command Line Developer Tools" → klik Install dan tunggu selesai.
3
Alternatif: Install via Homebrew (kalau pop-up tidak muncul)
Homebrew adalah "toko aplikasi" untuk Terminal. Install dulu Homebrew, lalu install Git lewat Homebrew.
# Tempel perintah ini ke Terminal, tekan Enter, tunggu selesai: /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)" # Setelah Homebrew selesai, install Git: brew install git
⏳ Sabar: Proses install bisa makan waktu 5–15 menit tergantung koneksi internet. Biarkan saja sampai selesai.
4
Perkenalkan dirimu ke Git (lakukan sekali seumur hidup)
Git butuh tahu siapa kamu. Ganti teks di dalam tanda kutip dengan nama dan email kamu yang asli:
% git config --global user.name "Nama Lengkap Kamu" % git config --global user.email "emailkamu@gmail.com"
Kalau tidak ada error dan tidak ada output apapun → itu berarti berhasil! (di Terminal, tidak ada output = sukses)
B · Buat Akun GitHub
1
Buka github.com
Buka browser Chrome → pergi ke github.com → klik tombol "Sign up" di pojok kanan atas.
2
Isi formulir pendaftaran
• Email: pakai email yang sama dengan yang kamu set di Git tadi
• Password: buat yang kuat dan simpan baik-baik
• Username: ini nama yang akan muncul di URL websitemu nanti (contoh: usman-dev). Pilih yang mudah diingat, huruf kecil semua, pakai tanda hubung (-) kalau perlu spasi.
3
Verifikasi email
GitHub akan kirim email verifikasi. Buka emailmu, cari email dari GitHub, klik link verifikasi. Akun kamu langsung aktif.
C · Hubungkan Terminal ke GitHub (satu kali setup)

Supaya Terminal di Mac-mu bisa "ngobrol" dengan GitHub, kita perlu buat kunci digital (SSH key). Ini seperti membuat cap tangan digital yang GitHub kenali.

1
Buat SSH key
Ketik di Terminal (ganti email dengan emailmu):
% ssh-keygen -t ed25519 -C "emailkamu@gmail.com"
Terminal akan tanya beberapa hal. Tekan Enter terus sampai selesai (kamu tidak perlu isi apa-apa).
2
Salin SSH key-mu
% pbcopy < ~/.ssh/id_ed25519.pub
Tidak ada output = SSH key sudah tersalin ke clipboard (seperti Ctrl+C).
3
Tambahkan ke GitHub
1. Buka github.com → klik foto profil kamu (pojok kanan atas) → Settings
2. Di menu kiri, klik "SSH and GPG keys"
3. Klik tombol hijau "New SSH key"
4. Title: isi bebas, misalnya "MacBook saya"
5. Key: klik di kolom kosong lalu tekan Cmd+V (paste)
6. Klik "Add SSH key"
4
Test koneksi
% ssh -T git@github.com
Kalau ditanya "Are you sure you want to continue?" → ketik yes lalu Enter.
Kalau muncul: Hi username! You've successfully authenticated → 🎉 berhasil!
✅ Checklist Sesi 1
Git terinstall (git --version menampilkan versi)
Nama dan email sudah di-set di Git
Punya akun GitHub dan sudah verifikasi email
SSH key sudah ditambahkan ke GitHub
ssh -T git@github.com menampilkan pesan "successfully authenticated"

A · Pahami dulu konsepnya

📦 Apa itu Repository?

Repository (repo) = folder proyekmu, tapi versi "pintar" yang menyimpan seluruh riwayat perubahan kode dari awal sampai sekarang. Analoginya seperti Google Drive, tapi khusus untuk kode dan bisa merekam semua edit yang pernah kamu buat.

Local repo = folder di laptop kamu
Remote repo = salinan folder itu di server GitHub (di cloud)
B · Buat Repository di GitHub
1
Buka github.com dan login
Klik tombol + di pojok kanan atas → pilih "New repository".
2
Isi detail repository
• Repository name: ketik proyek-cicd (huruf kecil, pakai tanda hubung)
• Description: boleh diisi bebas, contoh: "Belajar CI/CD pertamaku"
• Pilih Public (supaya bisa diakses via GitHub Pages nanti)
• JANGAN centang "Add a README file" (biarkan semua checkbox kosong)
• Klik tombol hijau "Create repository"
3
Halaman repo kosong akan muncul
GitHub akan menampilkan halaman dengan instruksi. Biarkan halaman ini terbuka — kita akan butuh URL-nya nanti.
🖥 Di halaman itu ada tulisan "Quick setup". Cari URL yang mirip: git@github.com:usernamekamu/proyek-cicd.git
C · Hubungkan folder lokal ke GitHub
1
Buka folder proyekmu di Terminal
Kita perlu masuk ke folder HTML yang sudah kamu buat sebelumnya. Perintah cd = "change directory" (pindah folder).
# Pindah ke folder proyek kamu # Ganti "nama-folder-kamu" sesuai nama folder proyekmu % cd ~/Documents/belajar-web # Kalau folder belum ada, buat dulu: % mkdir ~/Documents/belajar-web % cd ~/Documents/belajar-web
2
Buat file HTML sederhana (kalau belum ada)
Buka VS Code, buat file index.html di folder tersebut dengan isi minimal seperti ini:
<!DOCTYPE html> <html lang="id"> <head> <meta charset="UTF-8"> <title>Proyek CI/CD Pertamaku</title> </head> <body> <h1>Halo, ini website pertamaku dengan CI/CD!</h1> <p>Setiap kali saya edit dan push, website ini otomatis update.</p> </body> </html>
3
Inisialisasi Git di folder ini
Perintah ini "mengaktifkan" Git di foldermu — hanya dilakukan sekali per proyek.
% git init Initialized empty Git repository in /Users/.../belajar-web/.git/
4
Tambahkan semua file dan buat commit pertama
git add . = "tandai semua file untuk disimpan"
git commit -m "..." = "simpan snapshot dengan pesan catatan"
% git add . % git commit -m "commit pertama: tambah index.html" [main (root-commit) a1b2c3d] commit pertama: tambah index.html 1 file changed, 10 insertions(+)
5
Hubungkan ke GitHub dan push
Ganti USERNAMEMU dengan username GitHub kamu yang sebenarnya:
% git remote add origin git@github.com:USERNAMEMU/proyek-cicd.git % git branch -M main % git push -u origin main Enumerating objects: 3, done. Writing objects: 100% (3/3), done. Branch 'main' set up to track remote branch 'main' from 'origin'.
6
Verifikasi di GitHub
Refresh halaman GitHub repo kamu. File index.html sekarang harus muncul di sana! 🎉
✅ Berhasil! Kamu baru saja melakukan push pertamamu ke GitHub. Ini fondasi dari semua CI/CD — setiap push inilah yang nanti memicu otomasi berjalan.
D · Alur kerja sehari-hari setelah ini

Untuk sesi-sesi selanjutnya, setiap kali kamu edit kode dan mau kirim ke GitHub, cukup 3 perintah ini:

# 1. Tandai semua perubahan % git add . # 2. Simpan dengan pesan catatan (isi pesan sesuai apa yang kamu ubah) % git commit -m "deskripsi perubahan kamu" # 3. Kirim ke GitHub % git push
💡 Tips commit message: Biasakan tulis pesan commit yang jelas. Bukan "update" atau "fix" — tapi "tambah section About di halaman utama" atau "perbaiki warna tombol merah". Ini penting kalau nanti proyek kamu besar.
✅ Checklist Sesi 2
Repository "proyek-cicd" sudah dibuat di GitHub
file index.html sudah muncul di halaman GitHub repo
Coba edit index.html, lalu git add → commit → push → refresh GitHub (file harus berubah)

konsep YAML praktik GitHub Actions
⏱ 1.5 jam
✓
▼
A · Apa itu YAML?

📝 YAML = Bahasa instruksi yang mudah dibaca manusia

YAML (dibaca "yam-el") adalah format penulisan konfigurasi. GitHub Actions membaca file YAML untuk tahu apa yang harus dilakukan secara otomatis.

Analoginya: YAML itu seperti resep masak. Kamu tulis: "kalau ada tamu datang (push), panaskan oven (jalankan server), lalu panggang kue (deploy website)".

⚠️ YAML sangat sensitif spasi! Indentasi (spasi di awal baris) menentukan struktur. Selalu pakai SPASI (bukan Tab). 2 spasi per level indentasi.
B · Bedah struktur YAML baris per baris
# Baris dimulai dengan # adalah komentar (tidak dieksekusi) name: Workflow Pertamaku on: push: branches: [ main ] jobs: sapa-dunia: runs-on: ubuntu-latest steps: - name: Checkout kode uses: actions/checkout@v4 - name: Sapa dunia run: echo "Halo CI/CD!"

Penjelasan tiap bagian:

name: Workflow Pertamaku
Nama workflow-mu. Bebas diisi apa saja. Ini yang muncul di tab Actions GitHub.
on: push:
Kapan workflow ini dijalankan? Jawaban: setiap ada "push" ke GitHub.
branches: [main]
Khusus push ke branch bernama "main" saja yang memicu workflow ini.
jobs:
Daftar "pekerjaan" yang akan dijalankan. Bisa ada banyak pekerjaan sekaligus.
runs-on: ubuntu-latest
GitHub akan menyediakan server Ubuntu (Linux) gratis untuk menjalankan kode. Kamu tidak perlu sediakan server sendiri!
steps:
Urutan langkah di dalam satu pekerjaan. Dijalankan dari atas ke bawah.
uses: actions/checkout@v4
"Pinjam" aksi siap pakai dari GitHub. Aksi ini mendownload kode kamu ke server GitHub.
run: echo "..."
Jalankan perintah Terminal biasa. echo = print/cetak teks.
C · Buat workflow pertama langkah demi langkah
1
Buat folder .github/workflows/ di proyekmu
Di Terminal (pastikan sudah di folder proyekmu):
% mkdir -p .github/workflows
Tanda titik di depan .github artinya folder tersembunyi. Di Finder tidak akan terlihat, tapi di Terminal dan VS Code bisa terlihat.
2
Buat file workflow di VS Code
Buka VS Code → buka folder proyekmu → di panel kiri, klik folder .github → klik folder workflows → klik ikon "New File" → beri nama halo-dunia.yml
3
Salin isi file ini persis
name: Workflow Pertamaku on: push: branches: [ main ] jobs: sapa-dunia: runs-on: ubuntu-latest steps: - name: Checkout kode uses: actions/checkout@v4 - name: Tampilkan pesan run: echo "Halo! Workflow CI/CD pertamaku berjalan!" - name: Tampilkan tanggal run: date
⚠️ Perhatikan spasi! Pastikan indentasi persis sama seperti contoh di atas. Kalau ada tab atau spasi yang salah, workflow akan error.
4
Simpan, commit, dan push
% git add . % git commit -m "tambah workflow pertama" % git push
5
Lihat workflow berjalan di GitHub!
1. Buka halaman GitHub repo-mu
2. Klik tab "Actions" (di menu atas, antara Pull requests dan Projects)
3. Kamu akan lihat workflow dengan nama "Workflow Pertamaku" sedang berjalan (ikon kuning berputar) atau sudah selesai (ikon hijau centang)
4. Klik nama workflow → klik "sapa-dunia" → expand step "Tampilkan pesan"
5. Kamu akan lihat output: Halo! Workflow CI/CD pertamaku berjalan!
🎉 Selamat! Kamu baru saja membuat CI/CD pertamamu! Setiap push ke branch main akan otomatis menjalankan instruksi di file YAML ini.
✅ Checklist Sesi 3
Folder .github/workflows/ sudah ada di proyekmu
File halo-dunia.yml sudah ada di dalamnya
Di tab Actions GitHub, workflow tampil dengan tanda centang hijau ✓
Bisa melihat output "Halo! Workflow CI/CD pertamaku berjalan!" di log Actions

A · Konsep CI (Continuous Integration)

🧪 Kenapa perlu pengecekan otomatis?

Bayangkan kamu punya klien dan website-nya sudah live. Kamu edit kode → lupa tutup satu tag HTML → push → website rusak di depan klien. Malu kan?

CI memastikan setiap kali kamu push, kode di-cek otomatis dulu. Kalau ada error, CI akan memberi tahu kamu sebelum kode masuk ke website.

💡 Analoginya seperti quality control di pabrik. Setiap produk dicek sebelum dikirim ke pelanggan.
B · Buat workflow CI untuk validasi HTML
1
Buat file baru: cek-html.yml
Di VS Code, buat file baru di folder .github/workflows/ bernama cek-html.yml
2
Isi dengan workflow CI berikut
name: Cek Kualitas HTML on: push: branches: [ main ] jobs: validasi-html: runs-on: ubuntu-latest steps: - name: Ambil kode dari repo uses: actions/checkout@v4 - name: Setup Node.js uses: actions/setup-node@v4 with: node-version: '20' - name: Install HTML Validator run: npm install -g html-validate - name: Jalankan validasi HTML run: html-validate "**/*.html"
3
Push dan lihat hasilnya
% git add . % git commit -m "tambah CI untuk validasi HTML" % git push
Buka tab Actions di GitHub → lihat workflow "Cek Kualitas HTML" berjalan. Kalau HTML-mu bersih → hasilnya hijau ✓
C · Eksperimen: Sengaja buat error lalu lihat CI menangkapnya!
1
Rusak file HTML-mu dengan sengaja
Buka index.html di VS Code → hapus salah satu tag penutup, misalnya hapus </h1> atau </body>
2
Push perubahan yang "rusak" ini
% git add . % git commit -m "uji coba: HTML sengaja rusak" % git push
3
Lihat CI mendeteksi error!
Di tab Actions → workflow akan merah ✗. Klik untuk lihat detail error. GitHub juga akan kirim email notifikasi bahwa ada yang gagal.

Sekarang perbaiki HTML-mu (kembalikan tag yang dihapus), push lagi → CI kembali hijau.
🎉 Inilah CI! Kode yang rusak ketahuan sebelum masuk ke production/live. Di dunia profesional, ini mencegah ratusan bug mencapai pengguna.
✅ Checklist Sesi 4
Workflow cek-html.yml sudah berjalan (hijau) di tab Actions
Sudah coba sengaja rusak HTML → CI menampilkan error (merah)
Sudah perbaiki HTML dan CI kembali hijau

CD: Deploy Otomatis ke GitHub Pages
Push kode → website langsung live di internet dalam 2 menit!
praktik CD GitHub Pages deploy
⏱ 1.5 jam
✓
▼
A · Aktifkan GitHub Pages di setting repo
1
Buka Settings repo
Di halaman GitHub repo kamu → klik tab "Settings" (paling kanan di menu atas repo, ada ikon gear).
2
Cari menu Pages
Di sidebar kiri, scroll ke bawah → klik "Pages" (di bawah bagian "Code and automation").
3
Ubah Source ke GitHub Actions
Di bagian "Build and deployment" → bagian "Source" → klik dropdown yang bertuliskan "Deploy from a branch" → pilih "GitHub Actions".
🖥 Setelah pilih "GitHub Actions", tampilan akan berubah dan menawarkan beberapa template. Kita akan pakai workflow sendiri, jadi abaikan template tersebut.
B · Buat workflow deploy otomatis
1
Buat file baru: deploy.yml
Di VS Code, buat file baru di .github/workflows/ bernama deploy.yml
2
Isi dengan kode berikut persis
name: Deploy ke GitHub Pages on: push: branches: [ main ] permissions: contents: read pages: write id-token: write concurrency: group: "pages" cancel-in-progress: false jobs: deploy: environment: name: github-pages url: ${{ steps.deployment.outputs.page_url }} runs-on: ubuntu-latest steps: - name: Ambil kode uses: actions/checkout@v4 - name: Setup Pages uses: actions/configure-pages@v4 - name: Upload semua file website uses: actions/upload-pages-artifact@v3 with: path: '.' - name: Deploy ke GitHub Pages id: deployment uses: actions/deploy-pages@v4
3
Push dan tunggu deploy
% git add . % git commit -m "setup CD: deploy otomatis ke GitHub Pages" % git push
4
Tunggu 1–3 menit, lalu buka website-mu!
1. Buka tab Actions di GitHub → tunggu workflow "Deploy ke GitHub Pages" selesai (hijau)
2. Klik workflow tersebut → di bagian bawah ada link URL website-mu
3. URL-nya akan berbentuk: https://USERNAMEMU.github.io/proyek-cicd/
4. Atau cek di Settings → Pages → di bagian atas ada link "Your site is live at..."
🎉 Website kamu LIVE! Sekarang siapapun di seluruh dunia bisa membuka URL tersebut dan melihat website-mu. Dan setiap kali kamu push perubahan, website otomatis update dalam 1–2 menit!
C · Uji coba: edit → push → lihat perubahan live
Ubah teks di index.html (misal: ubah warna background atau tambah paragraf baru)
git add → commit → push
Tunggu 1–2 menit → refresh URL website → perubahan terlihat otomatis!
Share link website-mu ke seseorang — ini website pertamamu yang live di internet!

A · Konsep Branch

🌿 Branch = "cabang" paralel dari kodemu

Bayangkan kamu punya website yang sudah live dan diakses klien. Kamu mau tambah fitur baru, tapi takut merusak yang sudah ada.

Solusinya: buat branch baru. Kamu bebas bereksperimen di branch ini tanpa menyentuh branch main (yang live). Setelah fiturnya siap dan sudah di-cek, baru digabungkan ke main.

🌳 Branch main = versi website yang sudah live
🌿 Branch fitur-baru = area eksperimen, aman untuk dicoba-coba
B · Perintah-perintah branch
# Lihat branch yang ada (tanda * = kamu sedang di branch ini) % git branch * main # Buat branch baru sekaligus pindah ke sana % git checkout -b tambah-footer # Kamu sekarang di branch "tambah-footer" # Edit kode, commit seperti biasa... % git add . % git commit -m "tambah footer di halaman utama" # Push branch ini ke GitHub % git push origin tambah-footer # Kembali ke branch main % git checkout main
C · Pull Request: minta izin sebelum merge
1
Setelah push branch, buka GitHub
GitHub biasanya langsung menampilkan banner kuning: "tambah-footer had recent pushes" dengan tombol "Compare & pull request". Klik tombol itu.
2
Isi detail Pull Request
• Title: tulis apa yang kamu ubah, contoh: "Tambah footer dengan link sosmed"
• Description: jelaskan lebih detail kalau perlu
• Klik "Create pull request"
3
Lihat CI otomatis berjalan di PR!
Di halaman Pull Request, scroll ke bawah. Kamu akan melihat workflow CI (cek HTML) sedang berjalan otomatis untuk branch ini. GitHub tidak akan membiarkan merge kalau CI gagal.
✅ CI lolos? Klik tombol hijau "Merge pull request" → "Confirm merge". Branch-mu sekarang bergabung ke main, dan CD otomatis menjalankan deploy ke website!
4
Hapus branch yang sudah di-merge
GitHub akan menampilkan tombol "Delete branch" setelah merge. Klik untuk membereskan. Di Terminal juga perlu update:
# Pindah ke main dan update ke versi terbaru % git checkout main % git pull # Hapus branch lokal yang sudah di-merge % git branch -d tambah-footer
✅ Checklist Sesi 6
Buat branch baru, edit sesuatu di HTML, push branch tersebut
Buat Pull Request di GitHub
Lihat CI berjalan otomatis di dalam PR
Merge PR → website live update otomatis!
